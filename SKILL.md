# threads-yunwei-va — VA 美容诊所专属 Threads 自动运营系统

> **Claude 不只是内容生成器，而是整个运营系统的决策大脑。**
> 从抓取数据到筛选目标，从生成内容到判断是否值得发，每一个节点都由 AI 做决策，不是规则引擎，不是关键词匹配。

---

## 前置条件：必须满足，否则拒绝启动

```
✅ Claude AI（claude-sonnet-4-6 或以上）正在运行并可调用
✅ Threads CLI 工具已安装（threads-skill）
✅ Chrome 已启动调试模式
✅ Threads 账号已登录
```

**AI 是本系统的核心引擎，不是辅助工具。**
若 Claude 无法正常运行，本 skill 的所有筛选、判断、生成能力全部失效，退化为无意义的机械脚本。在此情况下必须终止，不得继续执行。

## AI 驱动架构总览

```
数据输入  →  AI 分析判断  →  执行  →  AI 评估结果  →  AI 更新规则
   ↑                                                         ↓
   └─────────────────── 闭环反哺 ────────────────────────────┘
```

**每个阶段 AI 的具体职责：**

| 阶段 | 机械工具做什么 | AI（Claude）做什么 | AI必须介入 |
|------|-------------|-----------------|----------|
| 自检 | 无 | 声明就绪、加载记忆、确认角色 | ⛔ 无AI不启动 |
| 抓取 | 执行 CLI 命令，获取原始数据 | 无 | — |
| 数据分析 | 无 | 读懂内容语义，识别话题，判断用户意图 | ⛔ 必须 |
| 选题决策 | 无 | 结合记忆+热度分，判断今日发什么 | ⛔ 必须 |
| 洗稿判断 | 无 | 判断值不值得跟，该从哪个角度切入 | ⛔ 必须 |
| 内容生成 | 无 | 写帖文，每条原创，不套模板 | ⛔ 必须 |
| **评论筛选** | 无 | **AI语境判断，从600条里筛出20-30条** | ⛔ 必须 |
| 评论生成 | 无 | 针对每条帖子定制评论，不重复 | ⛔ 必须 |
| 效果回溯 | 抓取帖子数据 | 分析为什么好/差，提炼规律 | ⛔ 必须 |
| 记忆更新 | 写入文件 | 决定写什么、写哪层、置信度多少 | ⛔ 必须 |
| 进化审查 | 无 | 诊断、生成假设、评估实验 | ⛔ 必须 |
| SKILL提案 | 无 | 识别规则与经验差距，起草修改方案 | ⛔ 必须 |

> **规则**：标注「⛔ 必须」的阶段，Claude 必须主动输出判断过程，不得静默跳过。
> 若某阶段 Claude 只是重复执行命令而没有做出判断，视为该阶段执行失败。


### 2.2 三维热度评分模型

对每个话题簇计算综合热度分（满分100）：

**维度一：互动数据分（40分）**
```
= (该话题帖子的平均点赞数 + 平均回复数×2 + 平均转发数×3) 的标准化得分
回复权重最高，因为回复代表主动参与，是真实热度的最强信号
```

**维度二：跨源验证分（35分）**
```
仅在 Feed 出现          → +10分
仅在关键词出现          → +8分
仅在对标账号出现        → +8分
Feed + 关键词 同时出现  → +20分
Feed + 对标账号 同时出现→ +22分
三源都出现              → +35分（满分，极强热点）
```

**维度三：时效性分（25分）**
```
帖子发布时间：
  0-6小时内   → 25.0分（满分，发酵中，参与价值最高）
  6-24小时    → 16.7分（正常）
  24-48小时   → 10.0分（热度衰减）
  48小时以上  →  3.3分（基本冷却）
```

**综合热度分 = 互动分×0.4 + 跨源分×0.35 + 时效分×0.25**

> 三维评分由 `filter-comment.py` 实际执行，不是 Claude 估算。

### 2.3 内容方向决策

根据评分结果，自动决定今日内容策略：

**发帖选题规则**：
- 选综合热度分 **Top 1** 的话题作为主帖方向
- 选综合热度分 **Top 2-3** 的话题作为备用方向（用于第二条帖）
- 若 Top 话题与本账号领域无关联性（如纯娱乐话题），顺延到下一个

**回复目标筛选规则**：

机械过滤（先排除明显不合适的）：
- 排除：已在 `list-replied` 返回列表中（threads-skill 内置防重复）
- 排除：政治/歧视/医疗事故敏感内容
- 排除：发帖时间 > 48小时（太旧）

AI 深度筛选（交由 Phase 4.1 执行，调用 `filter-comment.py` 完成）：
- `should_comment: true`  + `priority: high/medium` → 进入今日评论队列
- `should_comment: false` + 帖子热度高 → 点赞候选池
- `should_comment: false` + 帖子热度低 → 跳过

输出今日评论候选池（目标30-50条，执行时按 score_total 优先取20-30条）

- 结尾：自然带出账号特色，不超过2句，不用「私訊我」等硬引流词

**差异化素材来源**（AI 从以下位置提取，按需自然融入）：
- `ACCOUNT_PROFILE` 中配置的账号特色、从业背景、核心优势
- `KNOWLEDGE_FILES` 中的产品/服务知识、FAQ、品牌故事
- 不同于竞品的独特视角（通过对标账号分析获得）

**对标爆款借鉴规则**：
- 若今日对标账号有互动分 > 80 的爆款，分析其**结构和情绪调性**
- 用本账号的视角和内容重新演绎，不抄内容，借鉴形式

**禁止内容**（通用基准，可通过 KNOWLEDGE_FILES 追加账号专属禁区）：
- 虚假案例、夸大效果、具体价格承诺
- 政治相关、争议话题、直接攻击竞品

**价格表使用说明**（重要）：
- 若 `KNOWLEDGE_FILES` 包含价格表（如 `va-prices.md`），AI 可在发帖和评论时**自然使用价格信息**
- 价格对比型文案：台湾价格 vs 韩国 VA 价格（突出 3-5 倍价差）
- 项目推荐型文案：VA 招牌项目 TOP3 + 价格 + 优势
- 评论引流型文案：具体项目价格 + 台湾对比 + 省多少钱
- 价格单位：万韩元，可标注约等于台币（1 万韩元 ≈ 250 台币）
- 免责声明：价格仅供参考，具体价格因个人情况而异

### 即时洗稿协议（发现热帖30分钟内完成）

> 不是抄文字，是抢话题窗口。借热帖的流量势能，用自己账号的角度重新讲。

**第一步：解构热帖**
```
提取：话题是什么 / 为什么这条在爆（情绪点？信息缺口？争议？）
提取：开头句式结构 / 内容节奏 / 结尾方式
不要：对方的具体文字、案例、数据
```

**第二步：本账号角度重写**
```
同一话题，换成本账号 $ACCOUNT_PROFILE 视角切入
加入本账号独有素材（来自 ACCOUNT_PROFILE 和 KNOWLEDGE_FILES）
提供对方没有的专业深度或不同结论
字数控制在原帖 80-120% 之间
```

**第三步：时效性标记**
```
发布后在 yunwei_own_posts.json 标注 source: "trending_rewrite"
用于后续分析：跟车帖 vs 原创帖，哪类互动更好
```

**洗稿禁区**：
- 不复制原文任何完整句子
- 不使用对方帖子里的虚构案例
- 若原帖涉及医疗事故/负面新闻，不蹭，直接跳过

### 跟车发帖协议（对标账号爆款出现时）

当对标账号某条帖子互动异常高（>均值3倍），且发布时间<6小时：

```
分析其爆款结构 → 判断本账号是否能提供「专业延伸」视角
能 → 立即发一条「从本账号专业角度补充」的帖子
不能（内容无关） → 转为评论该帖，借其流量留下我们的存在感
```

### 内容去重检查

发布前，对照 `yunwei_own_posts.json` 中近**7天**的帖子主题：
- 若本次选题与近期帖子高度相似（同一话题+同一角度）→ 切换到备选方向
- 若仅话题相同但角度不同 → 可发，不同视角本身就是价值

### 发布执行

```bash
python scripts/cli.py --account "$THREADS_ACCOUNT" post-thread --content "[生成的帖文]"
```

发布成功后，立即将帖子URL和元数据写入 `yunwei_own_posts.json`：

```json
{
  "url": "https://www.threads.net/@va/post/xxx",
  "published_at": "YYYY-MM-DD HH:mm",
  "topic": "话题名",
  "content_type": "院长科普/防踩坑/...",
  "topic_score": 82,
  "actual_engagement": null,   ← 发布时为空，回溯时填入
  "rating": null               ← 回溯后评级

```

更新 `yunwei_session.json` 的 `published_urls` 和 `last_completed_phase: 3`。

若有第二条帖文，等待 **5分钟** 后发布（避免连发被降权）。

### 发帖后黄金窗口互动（发布后30分钟内）

> Threads 算法对帖子发出后头1小时的互动密度敏感。首批回复的质量直接影响推流范围。

发布后等待 **15分钟**，然后抓取自己帖子的回复：

```bash

```

**处理规则**：

- 若有人回复：Claude 以院长身份回复首条最有价值的留言（回复要短、有温度，≤50字）
- 若无人回复：点赞自己帖子（增加帖子互动基数）
- 等待再15分钟后重复检查一次（共检查2次，不超过发帖后45分钟）

**回复原则**：不主动推销，把评论者的问题接住，让对话继续——算法会因互动持续而扩大推流。

## PHASE 4：AI 驱动的精准评论引流

> ⛔ **此阶段每一条评论都必须经过 Claude 独立判断，禁止批量套用模板。**
> **评论是引流的核心武器，不是附属动作。**
> 每条评论都经 AI 严格筛选目标 + AI 定制内容，不发无意义的评论。
> 每日目标：**20-30条高质量引流评论**。

---

### 4.1 评论目标筛选（发评论前必须执行）

> ⛔ 这是整个系统中最关键的环节。
> 关键词过滤 + 三维热度评分 + AI 语境判断，全部由 `filter-comment.py` 执行。

#### 执行命令

将 Phase 2 采集的帖子存为文件，调用筛选脚本：

```bash
FILTER_SCRIPT=~/Desktop/threads-filter-comment/filter-comment.py
TMPDIR="/tmp/threads-filter"

# 三源模式（推荐）
python3 "$FILTER_SCRIPT" \
  --feed-file      "$TMPDIR/feed.json" \
  --keyword-file   "$TMPDIR/keyword.json" \
  --benchmark-file "$TMPDIR/benchmark.json" \
  --only-approved \
  > "$TMPDIR/result.json"

# 仅 Feed（快速模式）
python3 "$FILTER_SCRIPT" \
  --feed-file "$TMPDIR/feed.json" \
  --only-approved \
  > "$TMPDIR/result.json"
```

筛选逻辑（代码实现，非 Claude 估算）：
- **第一关**：排除词 + 政治词过滤（`exclude_keywords` 配置于 `~/.threads-filter-comment.json`）
- **第二关**：关键词匹配 → 标记 `priority: high/medium`
- **第三关**：三维热度评分 → `score_total`（互动×0.4 + 跨源×0.35 + 时效×0.25）
- **第四关**：AI 语境判断 → `ai_should_comment: true/false`

**判断结果处理**：
```
ai_should_comment: true  + priority: high   → 立即执行，最高优先
ai_should_comment: true  + priority: medium → 加入队列
ai_should_comment: false                    → 跳过，仅点赞
```

**找不到合适帖子时**：重新执行 Phase 2 抓取，最多重试 3 次。

### 4.2 AI 定制每条评论内容

每条评论都由 AI 根据目标帖子的具体内容单独生成，**禁止套用固定模板**。

**评论分两种战术，AI 根据帖子类型自动判断用哪种：**

#### 战术 A：专业钩子型（目标：引出追问）

适用：对方有具体困惑或问题

```
核心逻辑：
  给出真正有价值的专业判断
  + 在最关键的地方停下来
  → 让对方忍不住问：「那我的情况呢？」

写法感觉：
「這種術後情況通常有兩個原因，一個處理起來很簡單
另一個需要更仔細評估⋯⋯
（然后停在这里，不说是哪两个）」

← 信息缺口触发好奇心，比任何广告都有效
```

#### 战术 B：认知颠覆型（目标：让有意向的人主动找来）

适用：对方在考虑某项目，或有既有认知可以被刷新

```
核心逻辑：
  一个反常识的专业结论
  + 简短说明为什么
  + 自然带出可以继续了解的出口

写法感觉：
「很多人以為[常见误区]，但其實⋯⋯
（用 $ACCOUNT_PROFILE 中的专业背景背书说明）
想了解更多的話，我們在[地点/渠道]」

← 不说「私信我」，说地点或具体出口，更真实，有画面感
```

**评论硬性规则（AI 生成时必须遵守）：**
- 50-150字，超出截短
- 不出现「私訊我」「加LINE」「歡迎洽詢」
- 不出现「我們最好」「業界第一」等无法证实的自吹话术
- 每条评论必须包含至少一个具体的专业信息，不能全是套话
- 同一天内同类话术不重复使用超过3次（AI 自行检查）

---

### 4.3 最终评论组装（发送前必须按此格式）

AI 生成评论内容后，按以下结构组装最终发送内容：

```
[AI生成正文] + [署名落款（可选）]
```

> 官方账号发出的评论，已有账号名称和头像作为身份背书，无需 @自己。

**署名落款（`$PROMO_SIGNATURE`）**：读取 `PROMO_SIGNATURE` 配置，按需追加到评论末尾。若 `PROMO_SIGNATURE` 为空，则所有评论只发 AI 正文，不追加落款。

**组装规则**：
- 高优先（`priority: high`）帖子 → `AI正文 + $PROMO_SIGNATURE`（明确表明身份，方便对方找来）
- 中优先（`priority: medium`）帖子 → 仅 `AI正文`（专业分享为主，不主动推销）
- 低优先（`priority: low`）帖子 → 只点赞，不评论

**组装示例**（以某医美账号为例）：
```
這種術後情況通常有兩個原因，一個處理起來很簡單，另一個需要更仔細評估……
——[账号名] [身份]
[地址/特色一句话]
```

```bash
# 间隔 ≥ 5秒（评论比点赞风险更高，间隔要更长）
```

每条评论执行后写入 `yunwei_own_replies.json`：

```json
  "replied_at": "YYYY-MM-DD HH:mm",
  "priority": "high / medium / low",
  "tactic": "A / B",
  "promo_appended": true,
  "reply_engagement": null
}
```

> `reply-thread` 只返回 `{postId, success, message}`，不返回我方评论的 URL。子回复监控通过父帖 URL + 作者匹配实现（见账号成长监控章节）。

**每日目标：20-30条，按扫描时段分布**

| 时段 | 目标评论数 | 来源 |
|------|---------|------|
| 08:00 晨间 | 5-8条 | 晨间候选池 |
| 12:00 午间 | 3-5条 | 午间扫描新帖 |
| 18:00 傍晚 | 3-5条 | 傍晚扫描新帖 |
| 21:00 黄金时段 | 8-12条 | 当日互动最高帖集中投放 |

## PHASE 6：更新运营记忆文档

> 每次运营结束后，Claude 按保质期将洞察写入对应的分层记忆文件。
> 不是追加流水账，而是提炼规律、修正认知、更新经验库。

### 更新原则

- **有新发现才写**：没有超出已有经验的内容不重复记录
- **修正优于追加**：若新数据否定了旧经验，直接修改旧条目，加注日期
- **定量优于定性**：尽量记录具体数字而非模糊描述
- **保持精炼**：每个章节控制在关键条目内，不堆砌

### 三层写入规则

每次运营结束后，按保质期决定写入哪一层：

**写入 `daily.md`**（今日发现，3天后评估是否升级）：
- 今日热度 Top 话题及评分
- 某条回复的意外反应（好或坏）
- 抓取时观察到的平台异常现象
- 对标账号今日发了什么值得注意的内容

**写入 `weekly.md`**（需3天以上数据验证后才写）：
- 某话题连续出现≥3天，热度趋势稳定
- 某类内容连续多次表现明显好/差
- 关键词搜索质量的变化趋势
- 对标账号的内容模式变化

**写入 `longterm.md`**（C3/C4置信度，≥11次数据）：
- 经充分验证的内容类型效果排序
- 经校准后稳定的评分权重
- 季节性规律（需跨年数据）
- 平台运营基本规律

### 内容资产库更新（`content-assets.md`）

当一条帖子评级为「爆款」或连续2次「良好」时，提炼其**可复用结构**写入资产库：

```markdown
  中段：专业解释 + $ACCOUNT_PROFILE 背书
  结尾：「在我们这边，这一步是⋯⋯」（不超过一句）
禁忌：不用「大家注意」「重要提醒」等官方腔
```

## 自进化机制（Evolution Loop）

> 系统不只是"存经验"，而是主动检测自身衰退并提出改变。

### 触发条件（满足任一即触发）

- 连续**5天**平均互动数低于历史均值的 60%
- 连续**3天**发帖评级均为「普通」或「无效」
- **每日必做**（每次晨间运营结束后强制执行）

### 进化审查流程（每日执行，按数据量自动调整深度）

**数据量判断**：
```
运营天数 D < 3   → 执行「观察模式」：只记录，不生成假设
3 ≤ D < 7        → 执行「初步诊断」：基于现有数据找最明显的差异
D ≥ 7            → 执行「完整审查」：全量对比 + 假设生成 + 上次假设评估
```

**观察模式（D < 3）**：
```
今日互动总量：XX
高于/低于昨日：XX%
值得记录的现象：[描述]
→ 写入 daily.md，等待数据积累
```

**初步诊断（3 ≤ D < 7）**：
```
已有数据 D 天，当前最高效内容类型：[类型] 均值XX
当前最低效内容类型：[类型] 均值XX
初步假设（待验证）：[描述]
→ 写入 daily.md，明日继续跟踪
```

**完整审查（D ≥ 7）**：

Step 1 诊断，对比近3天 vs 更早数据：
```
发帖平均互动：XX → XX（变化幅度）
最高效内容类型：[类型] 均值XX
最低效内容类型：[类型] 均值XX
关键词命中质量：高/中/低
上次假设执行结果：[成立/不成立/进行中]
```

Step 2 生成今日假设（1条，精准，可验证）：
```
假设：「[具体描述]，预期效果[指标]提升≥15%」
验证方式：明日起执行，3天后评估
```

Step 3 评估（若有3天前的假设）：
- 成立（提升 ≥ 15%）→ 升级到 weekly.md，触发置信度计数
- 不成立 → 记入 longterm.md 踩坑，回滚，生成新假设

**进化历史**存储在 `weekly.md`（进行中实验）和 `longterm.md`（已结案实验）：
```markdown
## 进化实验记录

| 实验 | 假设 | 执行期 | 结果 | 结论 |
|------|-----|-------|------|------|
| #1 | 提升防踩坑内容占比 | 03-10~03-17 | 互动+23% | 成立，已固化 |
| #2 | 替换关键词 | - | - | 进行中 |
```

## 评分模型自校准机制

> 固定权重是别人的经验，自校准权重才是自己的数据。

### 校准时机：每满30天运行一次

基于 `yunwei_own_posts.json` 中有回溯数据的帖子，分析三个维度的预测力：

**计算每个维度与实际互动的相关性**：
```
相关性计算：
  选题时的「互动数据分」vs 帖子实际互动数  → 得到相关系数 r₁
  选题时的「跨源验证分」vs 帖子实际互动数  → 得到相关系数 r₂
  选题时的「时效性分」  vs 帖子实际互动数  → 得到相关系数 r₃

新权重 = 各维度相关系数归一化后的比例
例：r₁=0.6, r₂=0.3, r₃=0.1 → 新权重 = 60%/30%/10%
```

**权重变更规则**：
- 单次调整幅度不超过 ±10%（防止剧烈震荡）
- 最低权重下限 10%（每个维度保留基本话语权）
- 将新权重和调整理由写入 `longterm.md` 的「评分模型权重历史」表格

## 记忆置信度系统

> 1次数据得出的"规律"和30次数据得出的"规律"不应该被同等对待。

### 置信度评级

记忆文档中每条规律条目附加置信度标注：

```
[C1] 初步观察  — 数据点 <3，仅供参考，不作为强权重
[C2] 有限验证  — 数据点 3-10，可参考，适度影响决策
[C3] 充分验证  — 数据点 11-30，可信赖，作为主要权重输入
[C4] 高度可信  — 数据点 >30，规律稳定，高权重应用
```

示例：
```markdown
| 内容类型 | 平均互动 | 置信度 | 数据点数 |
|---------|---------|-------|---------|
| 防踩坑类 | 34.2 | [C3] | 15次 |
| 院长科普 | 18.6 | [C2] | 7次 |
| 季节时机 | 9.1 | [C1] | 2次 |
```

**在策略决策中**：仅 C3/C4 级规律作为强约束，C1/C2 级仅作为参考提示。

## OpenClaw 全自动托管配置

> 以下配置使 OpenClaw 能够完全无人值守地每日运行本 skill。

### 调度配置

```yaml
# openclaw 调度建议（初始值，系统会根据数据提案优化）
schedule:
  - "0 8 * * *"    # 晨间全量（必跑）
  - "0 12 * * *"   # 午间扫描
  - "0 18 * * *"   # 傍晚扫描
  - "0 21 * * *"   # 黄金时段互动爆发
timezone: "Asia/Taipei"        # 与目标受众时区保持一致，按需调整
skill: threads-yunwei
working_dir: $THREADS_SKILL_PATH   # 由 PHASE SETUP 写入的 threads-skill 根目录
```

### 评论周期控制

> 防止单次执行过度活跃，超出账号安全阈值后自动休息。
> 灵感来源：threads-filter-comment 的 cycle control 机制。

周期状态存储在 `/tmp/threads-yunwei-cycle.json`：

```json
{
  "cycle_start": "YYYY-MM-DD HH:mm",
  "comments_this_cycle": 8,
  "max_per_cycle": 12,
  "rest_until": null
}
```

**周期规则**：

| 时段 | 本次最多评论 | 超出后 |
|------|------------|-------|
| 08:00 晨间 | 8条 | 停止评论，继续其他操作 |
| 12:00 午间 | 5条 | 停止评论 |
| 18:00 傍晚 | 5条 | 停止评论 |
| 21:00 黄金时段 | 12条 | 停止评论，触发1小时强制休息 |

**连续超标检测**：若连续3次扫描评论量均达上限，自动增加间隔至 ≥8秒，下调单次目标20%，写入 `platform-changes.md` 作为风控信号。

### OpenClaw 退出信号约定

每次运营结束，将结构化状态写入 `yunwei_session.json` 供 OpenClaw 读取：

```json

  "exit_reason": "completed_normally",
  "skill_update_pending": true,   // 是否有待确认的SKILL提案
  "human_action_required": false  // 是否需要人工操作（如重新登录）
}
```

### 异常处理策略

| 异常类型 | 检测方式 | 自动处理 | exit_code |

| 阶段中断恢复 | session 残留 | 断点续传 | 0 |
